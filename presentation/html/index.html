~/scribe Slide 02 Why this exists

READMEs are the chronically deferred chore.

“I'll write it later.”

— every developer, every project

1. Templates are generic.

Cookiecutter, npm init, cargo new — same skeleton every time. They know nothing about your project.

2. Cloud AI collects code.

ChatGPT, Copilot, Cursor will write a README — but you ship your repo to a third party to get it.

3. Writing it by hand is tedious.

You remember everything; you just don't want to type it twice.

Three problems compound: templates are generic, cloud AI is a privacy concern, hand-writing is boring. Result: most repos ship with a title and a one-liner.

~/scribeSlide 05Two halves, one engine

How it works

A  Deterministic analyzer

  • Walks the directory
  • Counts files by language
  • Parses pyproject.toml, package.json, Cargo.toml, go.mod
  • Detects 35+ frameworks
  • Finds tests, CI, Docker, license, scripts
  • Renders a top-3-level file tree

Output: a typed RepoFacts snapshot. Pure reads, no network.

B  Generative LLM

  • Section-scoped prompts, not a mega-prompt
  • Strict system prompt forbids preambles
  • Streams tokens as they arrive
  • Eight sections: title, overview, features, tech stack, installation, usage, development, license
  • Deterministic blocks (badges, tree) bypass the LLM

Model: qwen2.5-coder:7b via Ollama. ~4.7 GB.

Two halves: a deterministic analyzer that does the actual reading, and a local LLM that writes the prose. Splitting them is what keeps the output honest — the model never invents a framework, because the analyzer hands it the list.
~/scribeSlide 06The decisive design choice

Why a local LLM.

Privacy

Source code stays on your machine. No SOC2 review, no NDA carve-out, no API key.

Cost

Pulling the model is a one-time 4.7 GB download. Every README after that is free.

Determinism

A specific model tag produces stable output over time. Cloud models drift silently.

$ ollama pull qwen2.5-coder:7b
One command. Then you are offline forever.
The single biggest design call: local model, not cloud. Three concrete reasons — privacy, cost, determinism. Cost is the least talked about but quietly the most important; a tool you use every week shouldn't have a meter running.

~/scribeSlide 08Honest analysis

Where Scribe falls short

Manifest-less repos
→
Sparser features section

Without pyproject.toml / package.json, the analyzer has less to work with.

Existing README present
→
Model echoes the prior text

Future fix: feed only the first paragraph, not the full prior README.

Concurrent CPU load
→
5 s → 12 s per section

Local inference shares the machine. Close Xcode and try again.

Already shipped

  • Strict system prompt that forbids LLM preambles and sign-offs.
  • Section-scoped calls so a slow section doesn't sink the whole output.
  • Deterministic badges + project tree — these never go through the model.
Honest about limits. Manifest-less repos produce thinner output. Existing READMEs sometimes get echoed back. Concurrent CPU load doubles the per-section time. All known, all documented.

Ollama server

Local

No database. No queue. The only network call is localhost.

Architecture is intentionally boring: three layers, all on the user's machine. The Ollama server is the only "external" service and it's also local. Nothing to deploy, nothing to scale, no cloud costs.
~/scribeSlide 10Under the hood

Engineering practices

Documentation 14 sections in English Intro, problem, SoTA, requirements, use cases, class diagrams, mockups, CLI catalog, testing, architecture, results, future work — plus a presentation cheat sheet.
Tests 22 passing Unit (analyzer, prompts, templates) + integration (generator with stub LLM). ~78% coverage.
Continuous integration GitHub Actions Ruff lint + pytest with coverage report on every push.
Packaging Editable install pip install -e . registers the scribe command from pyproject.toml.
Reproducibility Pinned model tag Default qwen2.5-coder:7b. Pinning the tag freezes the writing style across runs.
Everything around the AI matters as much as the AI: full English docs, real tests, CI, clean packaging, pinned model. The rubric is mostly about engineering, not about the model itself.

Future work

Diff-aware mode

pre-commit hook

4

Multi-doc agent

Beyond READMEs: CONTRIBUTING, SECURITY, architecture diagrams, narrative around an API reference.

Where it goes next. Diff mode is the obvious follow-up. Plugins for project-specific sections. pre-commit hook so docs stay in sync with code. Then expand beyond README to the whole doc surface.
